# Bug-Fix Audit: Voice, Review, RBAC, and Sandbox

## Scope and pre-fix diagnosis

This audit was written before implementation. The current working tree and application structure were preserved; no branch switch, refactor, or duplicate portal was used.

### Voice/form assistant

- Live request path: `VoiceAssistantPanel` sends the recognized or typed transcript through `CitizenAssistantContext` to `frontend/src/api/voiceApi.js`, which posts `message` to `POST /api/assistant/ask`. A TestClient trace confirmed the exact phrases reached the backend.
- The frontend deliberately used `{ auth: false }` for assistant calls, so a logged-in citizen token was not attached.
- The backend endpoint called the real `assistant_service.ask` path, but `field_detector.py` had no occupation or normalized knowledge intents and did not recognize `mandalam`, common mixed-language spellings, or Indic field names.
- The broad monthly-income alias `income` captured `Income Certificate validity`, producing an incorrect monthly-income answer.
- Every first response was prefixed by `_first_language_intro`, which asks which field needs help even when a known field had already been detected. This made distinct handler output look canned.
- The document handler is live and schema-backed. The validity/eligibility/status intents were not wired into this form-assistant route.
- Relevant files: `frontend/src/citizen-portal/components/VoiceAssistantPanel.jsx`, `frontend/src/citizen-portal/context/CitizenAssistantContext.jsx`, `frontend/src/api/voiceApi.js`, `backend/app/api/assistant_routes.py`, `backend/app/citizen_assistant/assistant_service.py`, `field_detector.py`, `guidance_engine.py`, and `language_helper.py`.

### RBAC and government application review

- Authentication itself works: seeded `officer@niyamguard.local` logs in with role `officer`, and the shared API client attaches its bearer token to the existing officer endpoints.
- Existing `/api/officer/...` review endpoints already accept `officer`, `reviewer`, and `admin` and reuse the service-portal review logic.
- The required `/api/government/applications...` endpoints do not exist, so the government portal contract returns 404 rather than reaching authorization or review logic.
- Frontend route authorization permits only `officer`/`reviewer` under government routes and only `citizen` under citizen routes. It therefore contradicts the intended admin super-role and can redirect a valid admin before an API call.
- Citizen application endpoints use `require_roles("citizen")` despite the service layer already allowing admin/reviewer/officer access where appropriate. Admin currently receives 403 on `GET /api/applications`.
- Permission matrix to enforce: citizen routes = `citizen` or `admin`; government/review routes = `officer`, `reviewer`, or `admin`; sandbox routes = `sandbox_admin` or `admin`; chatbot = any authenticated seeded role; public verification remains unauthenticated.
- Relevant files: `frontend/src/utils/authUtils.js`, `frontend/src/api/client.js`, `frontend/src/api/servicePortalApi.js`, `backend/app/api/service_portal_routes.py`, `backend/app/api/government_routes.py`, `backend/app/security/rbac.py`, and `backend/app/forms/service_portal_service.py`.

### Sandbox PDF and publish pipeline

- Circular create and PDF generation already retain department, circular number, title, affected service, rule key, old/new values, effective date, body, and disclaimer. The generated bytes are a real `%PDF` document and the authenticated GET endpoint returns `application/pdf`.
- The sandbox UI labels the final action `Export` and describes manual intake.
- `POST /api/sandbox/circulars/{id}/publish` is a compatibility stub that calls `export_for_manual_upload`, marks the endpoint deprecated, and explicitly says publishing is disabled. It never creates a government inbox document.
- The government inbox already supports sandbox records with a `government_document_id` and PDF URL, so the missing link is a scoped publish service that ingests the existing generated artifact and records delivery/audit state.
- Relevant files: `frontend/src/government-portal/components/VirtualGovernmentSandbox.jsx`, `frontend/src/api/sandboxApi.js`, `backend/app/api/sandbox_routes.py`, `backend/app/extraction/sandbox_circular_service.py`, `backend/app/demo/pdf_generator.py`, and `backend/app/demo/government_inbox_service.py`.

## Reproduced pre-fix results

- `mandalam lo em rayali?` -> no detected field and generic fallback.
- `occupation lo Imran Ali` -> no detected field and generic fallback.
- `Income Certificate validity entha?` -> monthly-income guidance.
- `GET /api/government/applications` and `/pending` -> 404 for officer and admin.
- `GET /api/applications` -> 403 for admin.
- Sandbox publish -> 200 response saying publish is disabled; no government delivery.

## Planned minimal fixes

1. Add one deterministic normalization layer and direct guidance for the required intents; stop adding the generic first-turn question to known intents; attach auth to assistant requests and allow citizen/admin.
2. Align frontend role routing, widen only the citizen endpoints required for the admin super-role, and add government aliases that call the existing service functions.
3. Replace only the publish stub with ingestion into the existing government inbox, retain the separate export endpoint, and update the existing sandbox button/copy.
4. Add the requested regression tests, then execute automated, API, browser, and one-by-one flow verification. Results will be recorded in `FINAL_ONE_BY_ONE_TEST_REPORT.md`.
